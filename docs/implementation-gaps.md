## PWA / SPA 構成

- [ ] ログイン後画面をPWA内画面として統一する。
  - 設計: ログイン、ダッシュボード、アカウント、共有プロフィール、QR表示、同期状態をPWAとして扱う。
  - 実装: 既存のPHPレンダリング画面が中心で、Service Worker登録はあるがPWA内画面としての統一UIにはなっていない。

- [ ] イベントQR作成SPAを実装する。
  - 設計: `/app/events/new` で通信状態、予約済みトークン状態、QRプレビュー、写真追加、後送信まで扱う。
  - 実装: `/dashboard` のフォームと `public/assets/app.js` でIndexedDB保存と同期を部分実装している。

- [ ] プロフィール編集SPAを実装する。
  - 設計: `/app/profiles` でプロフィール一覧、編集フォーム、画像アップロード、SNSリンク編集を扱う。
  - 実装: 一覧は `/dashboard` 内、作成は `/profiles/new`、既存プロフィールの参照・編集は `/profiles/{id}/edit` のPHP画面で扱っている。

- [ ] オフライン同期状態画面または同期状態パネルを実装する。
  - 設計: 未送信件数、同期中件数、失敗件数、再送、削除、QR表示、イベント内容確認を表示する。
  - 実装: `syncPending()` はあるが、同期状態の一覧UI、再送UI、削除UI、失敗理由表示はない。

## オフライン同期データ

- [ ] IndexedDBの保存データを設計項目に合わせる。
  - 設計: `client_request_id`, `share_event_token`, `created_at`, `retry_count`, `last_error`, `sync_status`, `location` を持つ。
  - 実装: `pendingShareEvents` は `public_token`, `profile_id`, `body`, `expires_in`, `photos`, `publish`, `synced_at` などに留まる。

- [ ] 同期失敗時の再試行状態を保存する。
  - 設計: 同期失敗時に再試行でき、失敗状態が分かる。
  - 実装: 失敗時は `console.warn` のみで、`retry_count` や `last_error` を保存していない。

- [ ] オフラインQR表示時の位置情報保存方針を合わせる。
  - 設計: コメント、写真、位置情報をIndexedDBへ保存し、通信復帰時にサーバへ同期する。
  - 実装: QR画面の位置情報保存はサーバ上の `event_id` がある場合だけ動く。未同期ドラフトQRでは位置情報を取得・保存しない。
